# EH13.3a Narrow-Screen Audit

## Purpose
- Record the narrow-screen and tablet audit required before responsive fallback implementation begins in `EH13.3b`.
- Keep the audit focused on the live admin shell, dashboard, queue surfaces, and table-heavy modules that were already touched by EH13.

## Audit Method
- Read the current admin layout, shared page-shell, shared list/filter section, and the active EH13 queue modules directly from the repo.
- Trace repeated responsive patterns such as sidebar-first stacking, multi-column hero shells, wrapping action rows, and table-only overflow behavior.
- Do **not** introduce new responsive fallbacks in this step. This document defines the implementation target for `EH13.3b`.

## Audited Surfaces
- Dashboard shell and sidebar navigation
- `/admin/dashboard`
- Shared admin page shell and action area
- Shared record-list interaction shell
- `/admin/follow-up`
- `/admin/routes`
- Table-heavy list modules already using the EH13 list pattern or adjacent admin reporting patterns

## Findings

### 1. Sidebar-first mobile flow adds scroll tax before primary content
- `src/app/(dashboard)/layout.tsx` renders the full branded sidebar before the page body on sub-`lg` widths.
- `src/features/admin/components/dashboard-chrome.tsx` stays single-column until `lg`, so narrow screens read: branding, signed-in card, long nav, footer actions, then actual page content.
- Risk: operators on tablets or narrow laptops must clear a long navigation block before reaching the active queue or form.
- `EH13.3b` target: move the sidebar into a compact or toggleable narrow-screen navigation pattern while preserving the current desktop sticky layout.

### 2. Shared list pattern still collapses into horizontal-scroll-only tables
- The current list baseline improves search and filtering, but the records still end in `overflow-x-auto` tables without a shared stacked fallback.
- Repeated in:
  - `src/features/customers/components/customer-list.tsx`
  - `src/features/meters/components/meter-list.tsx`
  - `src/features/readings/components/pending-reading-approvals.tsx`
  - `src/features/billing/components/approved-reading-bill-queue.tsx`
  - `src/features/payments/components/payment-history-list.tsx`
  - `src/features/reports/components/collections-payment-list.tsx`
  - `src/features/reports/components/receivables-list.tsx`
  - `src/features/routes/components/route-scoreboard.tsx`
- Adjacent admin tables outside the EH13 list shell still follow the same pattern:
  - `src/features/auth/components/staff-access-board.tsx`
  - `src/features/system-readiness/components/system-readiness-board.tsx`
  - `src/features/tariffs/components/tariff-list.tsx`
- Risk: horizontal scroll preserves data completeness but weakens scan speed, row comparison, and action discoverability below desktop width.
- `EH13.3b` target: add a shared dense-table fallback that converts high-priority columns and actions into stacked cards or row summaries below a chosen breakpoint.

### 3. Dashboard and page hero shells are still tall before the first actionable queue
- `src/app/(dashboard)/admin/dashboard/page.tsx` uses a large hero plus a secondary right rail of staff/context cards before the operational cards and workflow links.
- `src/features/admin/components/admin-page-shell.tsx` similarly stacks hero copy, action links, and multiple stat cards before page-local forms or queues on narrow screens.
- Risk: dashboard and module entry pages remain readable, but the first taskable control can sit too far below the fold on tablet-sized use.
- `EH13.3b` target: tighten hero density below `lg`, reduce vertical travel, and keep primary actions closer to the first viewport.

### 4. Follow-up cards remain responsive, but action priority weakens when buttons wrap
- `src/features/follow-up/components/follow-up-board.tsx` already uses card-based rows, which is better than table overflow on narrow widths.
- The service-action panels and bill cards still place multiple actions in wrapping button groups, especially where reset, stage-advance, statement, and printable-notice actions coexist.
- Risk: primary escalation actions can drop below secondary buttons or split across lines inconsistently on smaller widths.
- `EH13.3b` target: define a consistent stacked or reordered action treatment for urgent queue cards so the next required action stays visually first.

### 5. Route operations combine two dense forms with a table-led scoreboard
- `src/features/routes/components/route-operations-manager.tsx` groups creation and assignment tasks into paired forms, then `src/features/routes/components/route-scoreboard.tsx` returns to a wide ownership/coverage/pressure table.
- Risk: the page remains usable, but route setup and route review still ask narrow-screen users to move between tall form sections and horizontally constrained analytics.
- `EH13.3b` target: preserve the current route domain split, but add narrower fallbacks for the scoreboard and reduce form-density pressure where paired controls compete for vertical space.

## EH13.3b Implementation Targets
- Add a compact narrow-screen navigation pattern for the protected admin shell.
- Introduce a shared responsive fallback for dense data tables used by the EH13 list surfaces.
- Reduce hero and stat-card depth on dashboard and module entry shells below desktop width.
- Normalize queue-card action layouts so the primary next action stays first when space tightens.
- Apply the new fallback patterns first to the audited EH13 surfaces before wider consistency cleanup in `EH13.4` and `EH13.5`.

## Status
- `EH13.3a` audit is complete.
- `EH13.3b` responsive fallback implementation remains **blocked pending user validation** of this audit checkpoint.

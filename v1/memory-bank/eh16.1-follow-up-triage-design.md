# EH16.1 Follow-Up Triage Technical Design

## Purpose
- Define the first implementation slice of `EH16`.
- Keep the scope narrow enough to ship without weakening the validated EH15 assistant baseline or the existing EH5 follow-up workflow rules.
- Ground the design in the current `/admin/follow-up` module, not in a new parallel AI surface.

## Confirmed Product Decision
- `EH16 V1` is a **proposal-only follow-up triage worker**.
- The first worker surface is **`/admin/follow-up`**.
- The worker **must not execute any DWDS action directly**.
- OpenClaw, if adopted, stays behind a **protected server-side adapter**.

## Current Repo Baseline
- The protected follow-up route already exists at `src/app/(dashboard)/admin/follow-up/page.tsx`.
- Current queue assembly already derives:
  - overdue and outstanding balance state
  - follow-up stage
  - queue focus such as `NEEDS_REMINDER`, `NEEDS_FINAL_NOTICE`, `READY_FOR_DISCONNECTION`, `DISCONNECTED_HOLD`, and `MONITORING`
  - service-action visibility for disconnection or reinstatement review
- Server-authoritative follow-up mutations already live in `src/features/follow-up/actions.ts`.
- Follow-up status synchronization and overdue calculations already live in `src/features/follow-up/lib/workflow.ts`.

## V1 Goal
- Help staff decide **which overdue follow-up items deserve attention first** and **why**.
- Reduce operator scan time and decision friction on the existing follow-up queue.
- Produce auditable triage proposals without mutating bill, customer, or service state.

## V1 Non-Goals
- No mutation of `Bill.followUpStatus`
- No disconnection or reinstatement action
- No notice generation
- No broad transactional-data search outside the visible follow-up context
- No autonomous retries, background swarms, or multi-worker delegation
- No relocation of the feature into `/admin/assistant`

## User Experience

### Entry Point
- Add an AI triage entry point inside `/admin/follow-up`.
- Place it near the existing queue controls, not in the assistant popup.
- Label should reflect advisory scope, for example `Run AI Triage`.

### Run Scope
- The worker runs only on the current authorized follow-up dataset already available to the page.
- Optional initial scope limit:
  - current filtered queue only
  - or top N highest-pressure items from the current queue
- V1 should prefer deterministic bounded scope over full-queue open-ended runs.

### Output Shape
- Return a ranked list of follow-up items or accounts with:
  - recommendation summary
  - suggested next review step
  - reason summary
  - confidence or caution flag
  - source basis
- Output stays informational.
- Staff can dismiss or ignore the proposal.
- Staff must still use the existing follow-up controls manually if they choose to act.

### Staff Actions In V1
- `Run triage`
- `View proposal`
- `Dismiss proposal`
- `Review later`
- No `Approve and execute`

## Recommended UI Composition

### New Panel
- Add a lightweight admin-surface panel above or beside the current follow-up queue.
- Show:
  - last run timestamp
  - run status
  - scope summary
  - proposal count
  - dismiss state

### Proposal Presentation
- Present proposals as a ranked list, not as a chat transcript.
- Each proposal card should show:
  - customer name
  - account number
  - bill or follow-up context
  - current follow-up stage
  - days past due
  - outstanding balance
  - proposed triage label such as `Review first`, `Escalate review`, `Monitor`, or `Needs human judgment`
  - concise rationale

### Dismissal
- A dismissed proposal should remain in history but fall out of the active suggestion view.
- Dismissal is not a workflow mutation; it is only an operator decision about the proposal itself.

## Worker Logic Boundary

### Allowed Inputs
- Current follow-up queue records already visible to the signed-in user
- Existing derived fields already assembled on `/admin/follow-up`
- Approved workflow guidance from EH15 retrieval sources when needed for reasoning wording

### Disallowed Inputs
- Hidden admin/security data
- Unbounded cross-module customer search
- Raw secrets, environment values, or unrelated assistant sources

### Allowed Outputs
- Ranking
- Summaries
- Suggested next review step
- Confidence or caution signals
- Citations or source basis

### Disallowed Outputs
- Direct mutation commands
- Imperative execution payloads
- Hidden chain-of-thought
- Unsupported certainty where evidence is weak

## Server Architecture

### Placement
- Add EH16 code under a new follow-up-adjacent domain, preferably:
  - `src/features/automation/` for shared worker infrastructure
  - `src/features/follow-up/` for follow-up-specific triage logic and UI wiring

### Recommended Module Split
- `src/features/automation/lib/automation-policy.ts`
  - shared policy checks for proposal-only behavior
- `src/features/automation/lib/openclaw-adapter.ts`
  - server-side adapter boundary if OpenClaw is used
- `src/features/automation/lib/automation-store.ts`
  - read and write worker-run records
- `src/features/follow-up/lib/follow-up-triage.ts`
  - deterministic prompt assembly, scoring helpers, payload shaping
- `src/features/follow-up/actions.ts`
  - add proposal-only worker-trigger and dismissal actions
- `src/features/follow-up/components/`
  - add triage panel and proposal list components

### Request Flow
1. Authorized user opens `/admin/follow-up`
2. User triggers `Run AI Triage`
3. Server reuses current follow-up dataset or rebuilds an equivalent bounded dataset
4. Server applies EH16 policy checks
5. Server calls OpenClaw adapter or local orchestration layer
6. Server stores run and proposal records
7. UI refreshes and displays the proposal set

## Authorization Model
- Reuse current `followUp` module access boundaries.
- Only users already authorized for `/admin/follow-up` may trigger or view triage runs there.
- Proposal dismissal should require the same protected module access.
- No separate AI-only permission model is needed in V1 unless later product scope demands it.

## OpenClaw Boundary
- OpenClaw must not read the database directly.
- OpenClaw receives a bounded server-prepared payload.
- OpenClaw returns structured proposal data only.
- All auth, filtering, logging, and persistence remain inside DWDS server code.
- If OpenClaw is unavailable, the run should fail safely and visibly without degrading the rest of the module.

## Observability
- Persist:
  - actor
  - worker type
  - scope summary
  - latency
  - model/provider
  - proposal count
  - failure reason
- Show recent runs in the follow-up module so staff and admins can inspect usefulness.

## Evaluation
- Validate proposal quality against a fixed internal set of follow-up cases:
  - overdue but not yet reminded
  - reminder sent and ready for final notice review
  - disconnection-review candidates
  - disconnected accounts on hold
  - ambiguous cases needing human judgment
- Measure:
  - whether the top-ranked proposals match expected operator priority
  - whether rationale matches visible record facts
  - whether the worker avoids overstating certainty

## Implementation Order
1. Add EH16.1 schema and migration for run/proposal/review records
2. Add server-side automation policy and adapter boundary
3. Build follow-up triage payload assembly from the current queue dataset
4. Add proposal-only trigger action and storage
5. Add `/admin/follow-up` triage panel and proposal list
6. Add dismissal and run-history support
7. Add evaluation fixtures and telemetry review

## Exit Criteria
- `/admin/follow-up` exposes a bounded AI triage trigger
- Worker output is persisted and visible as ranked proposals
- Output is clearly advisory and cannot execute workflow mutations
- Dismissal and run history are auditable
- OpenClaw integration, if enabled, remains a protected server-side implementation detail
